# 나무위키 실검 이유 표시 브라우저 익스텐션 (namu-hot-reason)

## 1. 문제 정의

- 나무위키 메인에 노출되는 **실시간 검색어(실검)**는 키워드만 보여줄 뿐, *왜* 그 키워드가 급상승했는지에 대한 맥락 정보가 없음.
- 유저는 실검을 클릭해 해당 문서에 들어가도, 그 문서 자체로는 "지금 왜 이게 뜨는지"를 바로 알기 어려움 (사건/이슈/밈/방송 등이 문서 본문에 즉시 반영되지 않는 경우가 많음).
- 결과적으로 실검의 정보 가치가 반감됨.

## 2. 해결 아이디어

- 아카라이브의 **"나무위키 실검 알려주는 채널"** (https://arca.live/b/namuhotnow) 에는 유저들이 현재 실검이 왜 뜨는지를 설명하는 글이 실시간으로 올라옴.
- 이 채널의 게시글을 주기적으로 파싱해 "키워드 → 이유" 매핑을 구축.
- **해당 키워드의 나무위키 문서 페이지에 진입했을 때**, 문서 상단(또는 정해진 위치)에 "지금 이 문서가 실검에 오른 이유"를 배너/카드 형태로 띄움.
- 브라우저 익스텐션 형태(Chrome/Edge, 추후 Firefox)로 배포.

## 3. 목표 (MVP 범위)

1. 나무위키 문서 페이지(`https://namu.wiki/w/<문서명>`)에 진입 시, 해당 문서 제목이 현재 실검 목록에 있는지 판별.
2. 실검에 있다면 아카라이브 나무핫나우 채널에서 해당 키워드에 대한 "이유" 게시글을 찾아 매칭.
3. 매칭된 이유를 문서 본문 상단에 카드 UI로 노출 (출처 링크 포함).
4. 실검에 없거나 매칭 실패 시 아무것도 표시하지 않음 (조용한 실패).
5. 설정 창에서 on/off, 카드 표시 위치, 갱신 주기 조절.

## 4. 아키텍처 개요

```
[Namuwiki 문서 페이지 진입]
        │
        ▼
[Content Script]
   - 현재 문서 제목 추출
   - 실검 목록 확인 (문서에 실검 위젯이 있을 때 or 캐시 사용)
   - 실검 대상이면 background에 "이 키워드의 이유" 요청
        │
        ▼
[Background/Service Worker]
   ├─ arca.live/b/namuhotnow 게시글 fetch + 파싱
   ├─ 키워드 ↔ 이유 매칭 (정규화·부분일치·유사도)
   └─ chrome.storage 캐시 (TTL 기반)
        │
        ▼
[Content Script]
   - 문서 상단에 "실검 이유 카드" 주입 (출처 링크 포함)
```

- **Content Script**: 문서 페이지에서 제목 추출 → 실검 여부 판단 → background 요청 → 응답 도착 시 카드 삽입.
- **Background (Service Worker)**: CORS 회피 및 요청 통합 관리. 실검 목록/아카 게시글 주기적 polling (기본 5~10분). 메시지 기반 API 제공.
- **Storage**: `chrome.storage.local`에 `{keyword, reason, source_url, fetched_at}` 캐시, 실검 목록 캐시 (짧은 TTL).
- **Options Page**: 갱신 주기, 카드 위치(문서 상단/플로팅), on/off.

## 5. 데이터 파이프라인

1. 실검 목록 수집
   - 나무위키 API 또는 메인/문서 페이지에 포함된 실검 위젯 파싱.
   - 주기적으로 갱신하여 background에 캐시.
2. 아카 게시글 수집
   - `https://arca.live/b/namuhotnow` 목록 fetch (HTML).
   - DOMParser로 제목 + 본문 링크 파싱.
   - 필요 시 개별 게시글 본문 fetch → "이유" 요약 추출.
   - 제목 패턴 예: `[실검] <키워드> - <이유>` 등 → 정규식 + 휴리스틱.
3. 매칭
   - 현재 문서명을 정규화 (공백/특수문자 제거).
   - 실검 목록에 존재하는지 확인 → 존재하면 아카 게시글에서 같은 키워드 검색.
   - exact → 부분일치 → 토큰 유사도 fallback.
4. 렌더
   - 나무위키 본문 컨테이너 상단에 shadow DOM 기반 카드 삽입.
   - 카드 구성: "현재 실검 N위 · 이유 요약 · [원문 보기] 링크 · fetched_at 시각".

## 6. 기술 스택

- **Manifest V3** 기반 Chrome Extension.
- 언어: TypeScript.
- 번들러: Vite + `@crxjs/vite-plugin` (혹은 esbuild).
- 스타일: Shadow DOM + 최소 CSS (나무위키 스타일 충돌 방지).
- 테스트: vitest (파서·매처 단위 테스트).

## 7. 주요 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| arca.live HTML 구조 변경 | 파서 모듈화, 셀렉터 상수화, 실패 시 기존 캐시 fallback |
| arca.live 요청 차단/Rate limit | 요청 주기 제한(기본 5~10분), 실패 시 backoff, 캐시 우선 사용 |
| 키워드 매칭 미스매치 | 정규화 + 부분일치 + 토큰 유사도 3단계 fallback, 미매칭 시 조용한 실패 |
| 나무위키 문서 DOM 변경 | content script 셀렉터 상수화, 주입 실패 시 no-op |
| 실검 목록 획득 경로 불안정 | 실검 위젯/API 다중 소스화, 캐시로 완충 |
| 저작권/약관 이슈 | 원문 링크 필수 표기, 본문은 짧은 요약만 인용 |
| 오정보 위험 (아카 글이 부정확) | 카드에 "사용자 제보 기반" 문구 + 출처 링크 명시 |

## 8. 마일스톤

- **M0 — 스캐폴딩**: manifest, content script, 문서 페이지 제목 추출.
- **M1 — 실검 판별**: 현재 문서가 실검 대상인지 판별하는 로직 + 실검 목록 소스 확정.
- **M2 — 아카 파서**: arca.live 목록/게시글 파싱, 키워드/이유 추출.
- **M3 — 매칭 & 카드 주입**: 문서 상단에 shadow DOM 카드 렌더링.
- **M4 — 캐시/주기 갱신**: service worker 폴링 + storage TTL.
- **M5 — 옵션 페이지 & 배포**: 설정 UI, 아이콘, 스토어 업로드 준비.

## 9. 오픈 퀘스천

- 실검 목록을 가장 안정적으로 얻는 경로는? (공식 API / 메인페이지 / 문서 사이드 위젯)
- arca.live 게시글의 제목/본문 형식이 얼마나 일관적인지? (샘플 수집 필요)
- 동일 키워드에 여러 게시글이 있을 때 어떤 걸 고를지 (최신 / 추천 수 / 조회 수)?
- 카드 표시 위치: 문서 최상단 고정 vs 우측 플로팅 vs 사용자 선택.
- 모바일 나무위키(`m.namu.wiki`) 지원 여부 (MVP는 데스크톱만).
- 익스텐션에서 외부 요청 시 사용자 동의(프라이버시) 문구 필요 여부.
